## Зачем (два потолка)

Ближний: пользователь, запускающий установку в непустом месте (повторно,
после сбоя, после битого self-update), должен получить понятный диалог, а
не сырой git fatal — CEO встретил именно его. Дальний: та же ветка = штатное
восстановление инстанса, опора надёжности self-update канала.

## Мандат (DoD)

1. **Три случая точки входа** (детект по построению: framework/ =
   пересоздаваемая read-only копия; данные = qroky/ рядом, живые файлы):
   - (а) framework/ + данные рядом → диалог «переустановка: данные не
     трогаю, framework пересоздам [переустановить/обновить/отмена]»;
     «переустановить» = снос framework/ + свежий клон + идемпотентный
     прогон install-state сверху — живое (workdir, профиль, токен,
     привязка) не трогается; «обновить» = маршрут --apply-update;
     «отмена» = выход без следа.
   - (б) framework/ без данных рядом → «осиротевший клон, пересоздать?» →
     да = снос + чистая установка; нет = выход.
   - (в) чистого места → обычная установка (байт-в-байт нынешнее
     поведение — паритет харнесом).
2. **Финал uninstall** ссылается на этот путь: «для переустановки просто
   запусти install заново — данные останутся» (все 3 локали).
3. **Recovery-семантика:** сценарий «битый framework/ после прерванного
   self-update» проходит через (а)/(б) до рабочего состояния без потери
   живого.
4. **Харнес:** сценарии на (а)×3 ответа, (б)×2, (в)-паритет, recovery;
   ассерты фальсифицируемые + мутационная проверка (INFO-037) в verify.
5. **Локали:** все новые строки через L_*, en/ro/ru, «да/da/yes».
